# 第 6 课：把文档向量化，基于向量实现真正的语义搜索

> 背景：前端工程师转 agent 开发。前 5 课是「工具调用」这条线（tool/MCP/agent loop），这课进入**另一条线：RAG（检索增强生成）**。核心是让 agent 能「真正理解语义」地搜文档，而不是靠关键词匹配。

---

## 1. 核心认知

### 1.1 RAG 解决的核心问题：不是「快」，是「塞不下」

新手最常误解：RAG 是为了「加快检索」。**错。** 核心是「**文档太多塞不进上下文窗口**」这个硬约束。

| 问题                            | 全塞 prompt    | RAG 怎么解决       |
| ------------------------------- | -------------- | ------------------ |
| **塞不下**（文档 > 上下文窗口） | ❌ 报错/截断    | ✅ 只塞相关几段     |
| **贵**（每次重发全部）          | ❌ token 爆炸   | ✅ 只发检索到的几段 |
| **不准**（大海捞针）            | ❌ 注意力分散   | ✅ AI 只看相关片段  |
| **快**                          | ❌ 每次处理全部 | ✅ 只处理几段       |

「快」是连带好处，不是根因。「塞不下」才是——文档一大就走不通「全塞」这条路。顺带用语义匹配替代关键词匹配（搜「怎么部署」能找到写「上线流程」的文档）。

### 1.2 语义问题怎么变成几何问题

文本变成向量后，**「语义相近」变成了「向量夹角小」**。机器不懂语义，但它懂夹角——这就是「语义问题转几何问题」的本质。

- embedding 模型把文本映射到 1024 维空间，每段文本是一个点
- **方向 = 语义主题，长度 = 信息量**
- 语义相近的文本 → 方向趋同 → 夹角小 → 余弦相似度大

模型怎么学会的？训练时看海量文本，学一个东西：**「经常出现在相似上下文里的词，语义相近」**（distributional semantics，分布语义学）。「部署」和「上线」周围上下文像 → 模型把它们放到向量空间相近位置。维度是模型学的、**不可解释的**，人类只提供数据不提供定义。

### 1.3 RAG 的固有脆弱性：失败不可见

RAG 是多环节链路（embedding → 存库 → 检索 → 拼prompt → 生成），**任一环节静默失败，后面全废但表面正常**。最阴险的是 embedding 全零：检索全 NaN，但程序不报错、AI 还能基于乱片段生成「看似合理的错答」。**失败不可见**是 RAG 最该警惕的脆弱性。

## 2. 逐段拆解

### ① 导入与模型准备（第 1-29 行）

```js
import { ChatOpenAI, OpenAIEmbeddings } from "@langchain/openai";
import { Document } from "@langchain/core/documents";
import { MemoryVectorStore } from "@langchain/classic/vectorstores/memory";
```

三个核心角色：
- **`ChatOpenAI`**：对话模型，最后生成回答（L1 用过）。
- **`OpenAIEmbeddings`**：embedding 模型，文本→向量。**RAG 的灵魂**，和 ChatOpenAI 是两个不同模型——一个「理解语义变向量」，一个「生成文字」。
- **`Document`**：文档容器，装「正文 + 元数据」。
- **`MemoryVectorStore`**：内存向量库，存向量+原文，能算相似度。

**关键：模型配置分两套**——`model`（ChatOpenAI）走 `API_KEY`/`BASE_URL`/`MODEL_NAME` 生成回答；`embeddings`（OpenAIEmbeddings）走 `EMBEDDINGS_*` 向量化。因为生成模型和 embedding 模型通常不是同一个（这里生成用 Qwen，embedding 用 bge-m3），接口/key/计费都不同。

### ② `encodingFormat: "float"`（第 27 行）—— 本课踩的坑

```js
const embeddings = new OpenAIEmbeddings({
  apiKey: process.env.EMBEDDINGS_API_KEY,
  model: process.env.EMBEDDINGS_MODEL_NAME,
  configuration: { baseURL: process.env.EMBEDDINGS_BASE_URL },
  encodingFormat: "float",   // ← 关键修复
});
```

OpenAI embedding 接口支持两种返回格式：
- `"float"`（默认）：浮点数组 `[0.01, -0.02, ...]`
- `"base64"`：base64 编码二进制（省带宽）

**坑**：LangChain 的 `OpenAIEmbeddings` 默认请求带 `encoding_format: "base64"`，xx maas 接口对 base64 解析有问题，**返回 256 维全零向量**（静默兜底，不报错），导致相似度全 NaN。显式指定 `"float"` 让接口返回浮点数组，才正常（1024 维）。

### ③ 准备文档（第 31-95 行）

```js
new Document({
  pageContent: `光光是一个活泼开朗的小男孩...`,   // 正文，会被向量化、被检索
  metadata: { chapter: 1, character: "光光", type: "角色介绍", mood: "活泼" },  // 元数据，不向量化
}),
```

- **`pageContent`**：正文，向量化 + 检索的对象。
- **`metadata`**：元数据，**不向量化**，但跟着文档走，能过滤、展示、溯源（「这段来自第 2 章，角色是东东」）。

**metadata 常被新手忽略但极重要**：能做「只在第 3 章搜」「只搜角色介绍类」过滤，回答里能标注来源。没它检索结果就是「光秃秃文本」，没法溯源。

这里 7 段手写短文档，没「分块」这步。**真实场景文档很长，要先 chunk（切块）再向量化**——本课简化了。

### ④ 建向量库（第 97-100 行）

```js
const vectorStore = await MemoryVectorStore.fromDocuments(documents, embeddings);
```

一行干三件事：① 对每个 pageContent 调 embedding 得向量 ② 存「向量+原文+metadata」进内存 ③ 建索引能查了。`MemoryVectorStore` 是内存的，程序停了就没了，每次重跑要重新生成；生产用 Chroma/Milvus 持久化。

### ⑤ 检索（第 102-113 行）

```js
const retriever = vectorStore.asRetriever({ k: 3 });
const retrievedDocs = await retriever.invoke(question);
```

- `asRetriever({ k: 3 })`：包装成检索器，k=3 = 每次取最相关 3 段。
- `retriever.invoke(question)`：问题向量化 → 和库里每个向量比相似度 → 取最像 3 个。

**这里发生两次向量化**：建库时文档向量化（一次），检索时问题向量化（每次查询一次）。问题和文档用**同一个 embedding 模型**，在同一个空间，所以能比——这是 RAG 成立的前提。

`similaritySearchWithScore` 是额外拿相似度分数打印，和检索本身重复（实际项目二选一，这里为演示同时调）。

### ⑥ 生成回答（第 129-146 行）

```js
const context = retrievedDocs.map((doc, i) => `[片段${i+1}]\n${doc.pageContent}`).join("\n\n━━━━━\n\n");
const prompt = `你是一个讲友情故事的老师。基于以下故事片段回答问题...
故事片段:
${context}
问题: ${question}
老师的回答:`;
const response = await model.invoke(prompt);
```

三步：① 拼 context（3 段正文）② 拼 prompt（人设+规则+context+问题）③ 调 model。

**这就是 RAG 的「R」和「G」**：R（Retrieval 检索）= 第⑤步捞片段；G（Generation 生成）= 第⑥步基于片段生成。

**关键**：AI 回答依据是 `context`（检索片段），不是自己的记忆。prompt 里「如果故事中没有提到，就说没提到」很重要——**限制 AI 只基于检索内容答，防幻觉**。这是 RAG 控质量的核心手段。

## 3. 关键机制详解

### 3.1 RAG 全链路数据流

```
文档数组 ──向量化──► 向量库 ──检索──► 相关片段 ──拼prompt──► AI回答
   ↑                    ↑                              ↑
 分块+元数据        MemoryVectorStore            retriever取k=3
```

5 阶段：①准备模型 → ②准备文档 → ③建向量库 → ④检索 → ⑤生成回答。

**两次向量化**：建库时（文档→向量，一次）+ 检索时（问题→向量，每次）。同一模型、同一空间，才能比相似度。

### 3.2 为什么用余弦相似度而不是欧氏距离

|            | 余弦相似度                         | 欧氏距离               |
| ---------- | ---------------------------------- | ---------------------- |
| 看什么     | 方向（夹角）                       | 位置（直线距离）       |
| 受长度影响 | ❌ 归一化掉                         | ✅ 受影响               |
| 适合       | 文本（方向=语义主题，长度=信息量） | 图像（长度有绝对意义） |

**核心**：文本向量「方向 = 语义主题，长度 = 信息量」。一段「部署」和一段「我们要把系统部署到生产环境上线」语义几乎一样（方向一致），但长短不同（长度不同）。用欧氏会被长度骗（位置差远→判不相似），用余弦只看方向（判相似）。

**检索关心「讲的是不是同一件事」（方向），不关心「讲了多少」（长度）**，所以用余弦。这也是 `MemoryVectorStore.similaritySearchWithScore` 默认返回余弦相似度（越大越像，范围 -1~1）的原因——不用 `1 - score` 转。

### 3.3 k 值：召回率 vs 精度的平衡

| k 大小       | 召回率（找全） | 精度（找准） | 问题                                     |
| ------------ | -------------- | ------------ | ---------------------------------------- |
| 太小（k=1）  | 低             | 高           | 检索错了没冗余、信息不全                 |
| 太大（k=10） | 高             | 低           | 噪声淹没信号、浪费 token、分散 AI 注意力 |

**本质**：在「召回率」和「精度」之间平衡。k=3 是经验值——既保证冗余（防唯一那段检索错），又不塞太多噪声。具体看段长（段长 k 小）、问题复杂度（要综合多段 k 大）、窗口大小（窗口大能容忍大 k）。

**反直觉**：多给不一定好。AI 在噪声里挑答案能力有限，给 10 段（7 段无关）可能被带偏答更差。**精炼 3 段 > 啰嗦 10 段**。这就是 RAG 有 rerank 技术——先粗检索多取（k=20），再精排挑真正最相关 3 段。

### 3.4 embedding 全零坑：RAG 头号坑

**症状**：相似度全 NaN。
**根因**：embedding 接口返回全零向量，余弦 = `0·0/(√0·√0)` = `0/0` = NaN。
**为什么接口返回全零**：代理/网关在 embedding 失败时（超时、格式不兼容、限流）选择**静默兜底**返回全零，图「不崩」优先于「正确」。`encoding_format: "base64"` 不兼容就是典型——LangChain 默认带这个参数，xx接口解析失败返回全零。
**修复**：`encodingFormat: "float"` 显式指定浮点格式，或自定义 embedding 类直接 fetch 绕过 LangChain。

**最阴险**：全零不崩不报错，AI 还能基于乱片段生成「看似合理的错答」——**失败不可见**。

### 3.5 RAG 调试第一原则：先验证向量，再查逻辑

RAG 出问题，90% 不在「检索逻辑」（比相似度、取 top-k 很简单可靠），在「向量本身」（调外部接口，最易兼容性问题、最易静默失败）。看到 NaN/全零/结果异常，第一反应是「向量是不是全零/维度对不对/模型是不是同一个」，而不是「相似度算法写错了」。

排查链路：症状（NaN）→ 怀疑向量全零 → debug 脚本打印向量 → 确认全零 → 分诊（原始接口 vs LangChain）→ 锁定根因。

## 4. 概念辨析

### 4.1 RAG vs 全塞 prompt

|            | 全塞 prompt               | RAG                                       |
| ---------- | ------------------------- | ----------------------------------------- |
| 适合文档量 | 小（< 窗口 1/3 且不常变） | 中大（塞不下 / 常更新 / 要溯源 / 高并发） |
| 成本       | 每次重发全部（贵）        | 只发检索片段（省）                        |
| 准确性     | 看全局、跨段推理强        | 局部问答，检索质量决定一切                |
| 溯源       | 难                        | 能标注来源（metadata）                    |
| 复杂度     | 简单                      | 要建库、维护                              |

**判断**：文档 < 窗口 1/3 且不常变 → 直接塞；否则 RAG。**别看到文档就上 RAG，先量一量。** 本课 7 段小故事（<1000 字）其实直接塞更准，用 RAG 是为教学。

### 4.2 ChatOpenAI vs OpenAIEmbeddings

|                | ChatOpenAI                                   | OpenAIEmbeddings                                             |
| -------------- | -------------------------------------------- | ------------------------------------------------------------ |
| 干啥           | 生成文字回答                                 | 文本→向量                                                    |
| 输入           | 文本                                         | 文本                                                         |
| 输出           | 文本                                         | 向量（数字数组）                                             |
| 配置           | API_KEY/BASE_URL/MODEL_NAME                  | EMBEDDINGS_API_KEY/EMBEDDINGS_BASE_URL/EMBEDDINGS_MODEL_NAME |
| 本课角色       | 第⑥步生成回答                                | 第③④步向量化                                                 |
| 是不是同一模型 | 通常不是（生成用 Qwen，embedding 用 bge-m3） |                                                              |

### 4.3 encoding_format: float vs base64

|            | float          | base64            |
| ---------- | -------------- | ----------------- |
| 返回       | 浮点数组       | base64 编码二进制 |
| 体积       | 大             | 小（省带宽）      |
| xx接口兼容 | ✅ 正常 1024 维 | ❌ 返回全零        |

## 5. 踩坑提醒

1. **embedding 全零不报错** —— 接口静默兜底返回全零，相似度全 NaN 但程序不崩。RAG 头号坑，先验证向量非零。
2. **`encodingFormat: "float"`** —— xx maas 对默认 base64 格式解析有问题返回全零，显式指定 float 才正常。
3. **生成模型和 embedding 模型分两套配置** —— 不是同一个，接口/key/计费都不同，别混用。
4. **问题和文档必须用同一个 embedding 模型** —— 否则不在同一空间，比相似度没意义。
5. **`similaritySearchWithScore` 返回的是余弦相似度** —— 越大越像，不用 `1 - score` 转（那是把距离转相似度才用的）。
6. **metadata 别忽略** —— 不向量化但能过滤、溯源，没它检索结果是光秃秃文本。
7. **k 不是越大越好** —— 多给会塞噪声、分散 AI 注意力，精炼几段胜过啰嗦一堆。
8. **prompt 要限制「只基于检索内容答」** —— 防 AI 用自己记忆幻觉，加「没提到就说没提到」。
9. **MemoryVectorStore 是内存的** —— 程序停了就没，生产用 Chroma/Milvus 持久化。
10. **RAG 失败不可见** —— 多环节链路任一处静默失败后面全废但表面正常，生产要加监控（维度/非零比例/相似度分布/是否引用检索内容）。

## 6. 和前面课程的关系

```
L1-L5  工具调用线（tool/MCP/agent loop）   agent「能做事」
   ↓
L6  RAG 线（检索增强生成）                 agent「能查知识」
```

前 5 课让 agent 能调工具做事（读文件、跑命令、操作浏览器），这课让 agent 能**基于自己的文档库回答问题**——不是靠模型训练时的记忆（会过时、会幻觉），而是靠实时检索最新文档。两条线最终会合：agent 既能调工具，又能 RAG 查知识，就是完整的智能体。

**RAG 和工具调用的本质区别**：工具调用是「让模型调外部函数拿数据」，RAG 是「先检索文档片段塞进 prompt」。前者模型主动调，后者程序先检索。两者常组合使用。

## 7. 下一步

- **加分块（chunk）**：真实文档很长，用 `RecursiveCharacterTextSplitter` 切成段再向量化
- **加持久化向量库**：换 Chroma，数据落盘不用每次重建
- **加 rerank**：先粗检索 k=20，再用模型精排挑 3 段，提精度
- **加 metadata 过滤**：实现「只在第 3 章搜」「只搜角色介绍类」
- **加可观测性**：监控 embedding 维度/非零比例、相似度分布、AI 是否引用检索内容
- **RAG + 工具调用组合**：agent 先 RAG 查文档，查不到再调工具（如查数据库）

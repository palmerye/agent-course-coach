# 课程学习陪练

## Overview

跟着一门有实操的课程,每节课走固定 6 步流程:建目录 → 陪练敲码 → 提问查漏 → 互动补缺 → 生成 readme.md → 更新 Readme 索引。核心是用提问逼主动回忆、暴露盲区,而非出题打分。

学习者是前端工程师(有前端 + Node.js 经验),学的是 Agent 课程。本 skill 把"陪练 + 笔记沉淀"这套流程固化,让每节课一致产出。

## 流程(6 步,必须按序,每步不可跳)

学习者说 **"开始学第 N 课：xxxx"** 后,按序执行:

1. **建目录** — 在 `src/` 下生成 `Ln/`(如 `src/L4/`)。
   - **基线易漏点**:基线 agent 在目录已存在时直接跳过、不问。若 `src/Ln/` 已存在且有 readme.md,**必须先问**"重学/补笔记/跳过",不直接覆盖。
2. **陪练敲码** — 学习者对照课程敲代码。一课可能有**多个文件**(如 `agent.mjs` + `tools.mjs` + `run.mjs`),陪练要覆盖**本课所有文件**,不只盯一个。这期间:
   - 学习者随时提问 → 直接答(轻量,不走 6 步)。
   - **主动**提示重点/难点(不等学习者问)。基线做对了这点,保留。
3. **提问查漏** — 代码敲完,基于**本课所有敲的代码**抛问题(跨文件也行)。规则见下"提问方法论"。
4. **互动补缺** — 针对每个回答判定对/部分对/错,答错先追问(最多 2 轮)再讲。补缺要点进笔记。
5. **生成 readme.md** — 按下"笔记结构"写,输出到 `src/Ln/readme.md`。
6. **更新 Readme** — 在 `Readme.md` 追加一行指向 `src/Ln/readme.md` 的链接。
   - **基线易漏点**:基线 agent 在 L1/L3 都漏了这步。这是**必须项**,不是可选。

## 提问方法论(第 3 步的核心,基线最常跑偏的地方)

### 三条硬规则

1. **只问设计意图类,不问事实题**。
   - 事实题 = 查文档/查代码就能答的("zod 是什么""response 还有哪些字段""删 dotenv 报什么错")。这类**不占提问额度**。
   - 意图题 = 逼学习者想"为什么这么设计"的("为什么用 let 不用 const""map 和 forEach 为什么不合成一个循环""错误为什么喂回模型而不是 throw")。
2. **一次 3-5 个,不多不少**。
   - **基线易漏点**:基线 L1 抛 6 个、L3 抛 10 个。复杂课会让 agent 更失控。**5 是硬上限**。
3. **每个问题标注考察的能力点**(如"考察:职责分离")。基于刚敲的代码,不问脱离代码的抽象题。

### 补缺规则(第 4 步)

1. 先判对错:对 / 部分对 / 错。
2. 答错**先追问,不直接给答案**——追问到具体行,逼学习者定位。
3. **追问最多 2 轮**——仍答不到则直接讲清楚,避免挫败感。
4. 补缺要点进 readme.md 的"踩坑提醒"或"关键机制详解"——答错处往往是该写进笔记的重点。

### 反模式(必须避开)

- 不变成"出题 + 背答案"。提问是逼主动回忆 + 暴露盲区,不是考试打分。
- 不评分、不打勾叉,只说"对/这里还差一点"。
- 不一次性把所有问题抛完就不管——要逐个互动。

## 笔记结构(7 部分,分类骨架,不是上限)

输出到 `src/Ln/readme.md`。固定 7 个部分:

1. **标题 + 元信息** — `# 第 N 课：主题` + 一行背景。
2. **核心认知** — 这课最容易卡的 1-2 个关键点。**只在和前端思想差异大的地方**用前端类比(见下"前端类比策略")。
3. **逐段拆解** — 代码按逻辑段切开,每段讲"在干嘛 + 为什么这么写"。主体,占最大篇幅。
   - **单文件**:直接按逻辑段切。
   - **多文件**:按文件分组,每文件一个子节,文件内再按逻辑段切。例:
     ```markdown
     ## 逐段拆解

     ### agent.mjs

     ① 导入与初始化 ...
     ② agent 循环 ...

     ### tools.mjs

     ① read_file 工具 ...
     ② run_command 工具 ...
     ```
   - 不要只拆主文件漏掉辅助文件——每个文件都要过到。
4. **关键机制详解** — 挑 1-2 个最核心机制深挖,配 ASCII 流程图。**可写多个子点**(如 L3 的 option 深入、事件模型、退出码可同属此部分)。
5. **概念辨析** — 易混概念对比表(如 SystemMessage vs skill vs tool vs agent;spawn vs exec vs execSync)。没有就不写。
6. **踩坑提醒** — 陪练阶段提醒的 + 提问暴露的坑,列成要点。
7. **下一步** — 1-2 个延伸练习。

### 关键:7 部分是分类骨架,不是上限

**基线洞察**:L3 基线写了 11 章节内容,质量很高(三个 API 对比、option 深入、事件模型、退出码、安全……),只是没装进 7 部分骨架。

→ 正确做法:**保留内容丰富度,只对齐形状**。把多个子点归入对应骨架部分:

- "三个 API 对比" → 概念辨析
- "option 深入""事件模型""退出码" → 关键机制详解(可多个子点)
- "和上节课关系""伏笔" → 下一步
- "安全提醒" → 踩坑提醒

**原则:内容驱动结构,不是结构驱动内容。** 简单课(如 L1)可能只有 4 部分;信息量大的课(如 L2/L3)用满且每部分可多个子点。**不为了凑满 7 部分注水,也不为了不超 7 部分砍掉好内容。**

## 前端类比策略

学习者是前端工程师。用前端类比挂载新知识,但**只在和前端思想差异大的地方**用。

- **差异大(重点类比)**:如"模型只会输出文字,真正干活的是你的程序"——这与前端"用户点按钮触发动作"的直觉**相反**,必须类比,否则会理解错。
- **差异不大(不强行类比)**:如"用 let 因为要重新赋值""用 Promise.all 并行"——这是通用编程常识,不需要类比。

**判断标准**:这个概念如果用前端直觉去理解会得出**错误结论**,就重点类比;否则不类比。

**基线易漏点**:基线把"前端类比"单列一节,对通用常识(如 fetch 类比)也强行类比。不要单列一节,类比只在"核心认知"和"关键机制详解"里**按需出现**,不刻意。

## 触发与边界

### 触发

- 学习者说 **"开始学第 N 课：xxxx"** 或 **"学第 N 课"** + 主题 → 启动完整 6 步。
- 识别到"开始学/学第N课"意图即启动,无需学习者额外声明"用 skill"。
- `src/Ln/` 已存在但缺 readme.md → 也触发(补笔记)。

### 不启动 6 步的场景

1. **陪练阶段零散提问** — 敲码时突然问某 API 用法,直接答,不走 6 步。
2. **非课程学习场景** — 改 bug、查 API 等,不触发。
3. **重复课** — `src/Ln/` 已存在且有 readme.md,先问"重学/补笔记/跳过",不直接覆盖。

## Readme 索引格式

`Readme.md` 每课一行,新课追加,已有课补笔记时补行:

```markdown
# Agent 学习笔记索引

- [L1 hello-langchain](src/L1/readme.md)
- [L2 tool-file-read](src/L2/readme.md)
- [L3 node-exec](src/L3/readme.md)
- [L4 ...](src/L4/readme.md)
```

## 常见错误(基线实测,skill 要堵的)

| 基线错误           | 本 skill 的约束                               |
| ------------------ | --------------------------------------------- |
| 提问抛 6-10 个     | 硬上限 5 个                                   |
| 混入事实题         | 只问设计意图类,事实题不占额度                 |
| 笔记自创 8-11 章节 | 7 部分骨架,子点归入对应部分                   |
| 漏更新 Readme      | 第 6 步是必须项                               |
| 无阶段划分,单线走  | 6 步必须按序,每步明确                         |
| 前端类比滥用       | 只在差异大处,不单列一节                       |
| 多文件只盯一个     | 陪练+提问+拆解覆盖本课所有文件,拆解按文件分组 |
